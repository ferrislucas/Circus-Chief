## Approach: Visual UI Verification via Playwright

Tests use **Playwright page interactions** combined with **CSS computed style evaluation** and **DOM inspection** to verify visual behaviors, responsive layouts, animations, and interactive components. Most tests seed data via API helpers and then verify UI behavior in the browser.

### Key Patterns

1. **Seed data** via helper functions (`seedProject`, `seedSession`, etc.)
2. **Navigate** via `navigateAndWait()` for reliable page loads
3. **Evaluate computed styles** via `page.evaluate()` for visual assertions
4. **Viewport manipulation** via `page.setViewportSize()` for responsive tests
5. **Keyboard simulation** via `page.keyboard.press()` for accessibility tests
6. **Toast triggering** via UI actions that produce toast feedback (duplicate, archive, delete, etc.)

## Critical Architecture Notes for Implementer

### Toast System
- **ToastContainer.vue** renders toasts from `uiStore.toasts[]`
- Each toast has: `id`, `type` (success/error/warning/info), `message`
- Auto-dismiss after `TOAST_DURATION` (5000ms) by default
- Manual close via `.toast-close` button
- Slide-in animation from right (`translateX(100%)`)
- Toast classes: `.toast`, `.toast-success`, `.toast-error`, `.toast-warning`, `.toast-info`
- Icon per type: checkmark (success), X mark (error), warning triangle (warning), info symbol (info)
- Container class: `.toast-container`

### Overflow Menu (OverflowMenu.vue)
- Trigger: `.btn-kebab` button with `aria-label="Session actions"` (on session detail)
- Menu items: Archive/Unarchive, Duplicate, Copy Session ID (optional), Delete (always last, separated by divider)
- Keyboard navigation: ArrowUp/Down, Enter, Escape, Tab
- States: `.is-highlighted` (keyboard/mouse hover), `.is-danger` (delete), `.is-disabled` (during deletion)
- Delete shows `.menu-item-spinner` when `isDeleting: true`
- Overlay: `.menu-overlay` covers screen when open
- ARIA: `aria-expanded`, `aria-haspopup="menu"`, `role="menu"`, `role="menuitem"`

### ActionMenu.vue (Generic)
- Similar structure to OverflowMenu but with configurable items
- Props: `items[]` (icon, label, action), `ariaLabel`, `triggerIcon`
- Emits: `action-click` event
- `data-testid` on menu items matches `item.action`
- Exposes: `isOpen`, `highlightedIndex`, methods (for component testing)

### Session Detail Loading States
- `.loading-state` with `.loading-spinner` â€” shown when `sessionsStore.loading` is true
- `.delete-overlay` with `.delete-spinner` â€” full-screen overlay during session deletion
- `.skeleton-list` with `.skeleton.card` â€” in SessionListView for archived sessions loading

### Responsive Design
- Desktop tabs: `.tabs-desktop` with `router-link` elements
- Mobile tabs: `.tabs-mobile` with `<select>` dropdown
- **Multiple breakpoints** (not a single mobile breakpoint):
  - `640px` â€” Global tab switching in `main.css` (`.tabs-desktop` â†” `.tabs-mobile` for SessionDetailView)
  - `480px` â€” SessionListView-specific tab switching and layout changes (`.tabs-desktop` â†” `.tabs-mobile` within `SessionListView.vue`)
  - `768px` â€” Session header layout, `.session-name` font-size changes: `1.25rem` (desktop) â†’ `1rem` (mobile) (in `SessionDetailView.vue`)
- Sticky tabs: `.tabs` with `position: sticky` + `top` offset matching header height
- **Important:** Setting viewport to 375px triggers ALL breakpoints. A viewport of 700px triggers the 768px breakpoint but NOT the 640px or 480px ones.

### Pagination / Load More
- Only for archived sessions in `SessionListView`
- `.load-more-container` with `.btn.btn-secondary` button
- Shows remaining count: "Load More (X remaining)"
- Disabled state shows "Loading..." text
- Driven by `sessionsStore.archivedPagination` (`total`, `offset`, `hasMore`, `loading`)

### Dark Mode
- **No Tailwind CSS** â€” the project uses pure CSS custom properties defined in `:root` (in `packages/web/src/assets/main.css`)
- Background: `--color-background: #0d1117` = `rgb(13, 17, 23)` (primary bg), `--color-background-soft: #161b22` = `rgb(22, 27, 34)` (cards/sections), `--color-background-mute: #21262d` (tertiary)
- Text: `--color-text: #c9d1d9` = `rgb(201, 209, 217)` (primary text), `--color-text-soft: #8b949e` = `rgb(139, 148, 158)` (secondary text)
- Border: `--color-border: #30363d`
- Accents: `--color-primary: #58a6ff` (blue), `--color-success: #3fb950` (green), `--color-warning: #d29922` (amber), `--color-error: #f85149` (red), `--color-info: #58a6ff` (blue, same as primary)

### Modal Dialogs
- Confirmation dialogs use native `window.confirm()` (intercepted in tests via `page.on('dialog')`)
- Custom modals have **two different class patterns**:
  - **ScheduleSessionModal:** `.modal-backdrop` + `.modal-content`
  - **TokenWeightsModal:** `.modal-overlay` + `.modal` with `.modal-header h2`
- Both custom modals support close-on-Escape and close-on-overlay-click behaviors

### Collapsible Sections
- Tool details use `<details>` / `<summary>` HTML elements (native collapsible)
- Other show/hide toggles (e.g., canvas trash) use Vue `v-if` but are simple visibility toggles, not animated collapsible sections

## Key Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Toast auto-dismiss timing is flaky due to 5s wait | Use `expect(toast).toBeHidden({ timeout: 7000 })` with generous buffer |
| Mobile viewport tests may not trigger CSS media queries reliably | Set viewport BEFORE navigation, verify with computed style checks. Note the three different breakpoints: 640px, 768px, 480px. |
| Loading state tests depend on API interception timing | Use `page.route()` with controlled delays, assert visibility before continuing |
| Pagination tests need 30 sessions (PAGE_SIZE=25) | Seed sessions in parallel with `Promise.all`, use 45s timeout |
| Dark mode color values may vary slightly by browser rendering | Assert RGB ranges rather than exact values (e.g., R â‰¤ 20 for `#0d1117` primary bg). CSS custom properties are defined as exact hex values in `:root`, so variation should be minimal. |
| Custom modal tests may be hard to trigger if modal UI requires complex setup | Choose the most easily accessible modal (e.g., ScheduleSessionModal) or skip if setup is prohibitive and note as a known gap |
